2. Code Review & Phân tích lỗ hổng

2.1 🟥 Lỗ hổng Bảo mật NGHIÊM TRỌNG

BUG-01: Toàn bộ API endpoint không có [Authorize]

⛔ CAUTION – Mức độ: CRITICAL
Mọi endpoint đều PUBLIC, bất kỳ ai cũng có thể gọi API quản lý bàn, xóa sản phẩm, tạo user, xem thông báo nội bộ.

Hiện trạng: JWT được cấu hình đầy đủ trong Program.cs (UseAuthentication + UseAuthorization), nhưng KHÔNG có controller nào sử dụng [Authorize] hay [Authorize(Roles = "admin")].

Ảnh hưởng:

  • Khách hàng có thể gọi DELETE /api/Products/1 để xóa sản phẩm
  • Bất kỳ ai có thể POST /api/Users để tạo tài khoản admin
  • Thông tin nội bộ (doanh thu, logs) bị lộ hoàn toàn

BUG-02: QR Token không được validate khi đặt hàng

Hiện trạng: Bảng order_table có cột qr_code_token, nhưng API POST /api/Orders chỉ validate tableId > 0 mà không kiểm tra token từ QR code.

Rủi ro:

  • ⚡ Giả mạo QR: Kẻ xấu chỉ cần biết tableId (1-8) là gọi được API đặt món cho bất kỳ bàn nào
  • ⚡ Đặt hàng từ xa: Không cần ngồi tại quán, chỉ cần gọi POST /api/Orders với tableId=3
  • ⚡ Thao tác chéo bàn: Người tại bàn 1 có thể đặt món vào bàn 5

BUG-03: Hardcoded JWT Secret Key

"Key": "YOUR_SUPER_SECRET_KEY_MIN_32_CHARS"

Trong appsettings.json, key này là placeholder. Nếu deploy production với giá trị này → ai cũng có thể forge JWT token.

BUG-04: Hardcoded Connection String trong DbContext

Trong FoodStoreDbContext.cs, OnConfiguring chứa hardcoded connection string (Server=.;Database=FoodStoreDB;Trusted_Connection=True). Đây là artifact từ DB-first scaffold, cần loại bỏ.

2.2 🟧 Bug Logic Nghiệp vụ

BUG-05: State Machine ở KitchenController có lỗi OldStatus

// KitchenController.cs dòng 96-103
order.Status = "ready"; // gán trước
await _context.OrderStatusHistories.AddAsync(new OrderStatusHistory {
    OldStatus = order.Status == "ready" ? oldOrderStatus : order.Status, 
    // ← BUG: order.Status đã = "ready" → luôn dùng oldOrderStatus
    // Nhưng oldOrderStatus có thể cũng sai nếu đã bị ghi đè ở block "processing" phía trên
    NewStatus = "ready",
});

OldStatus trong history có thể bị ghi sai giá trị.

BUG-06: Order status "processing" không có trong CHECK CONSTRAINT

-- Database SQL:
CHECK (status IN ('pending', 'preparing', 'ready', 'served', 'paid', 'rejected', 'cancelled'))

// Nhưng KitchenController gán:
order.Status = "processing";  // ← Không nằm trong constraint → SQL Error!

BUG-07: Payment không validate trạng thái hiện tại

  • Không check order.Status trước khi thanh toán → Có thể thanh toán đơn đang pending (chưa nấu xong)
  • Không check order.PaymentStatus → Có thể thanh toán lại đơn đã success
  • ExpirePayment không kiểm tra trạng thái payment hiện tại

BUG-08: Không trừ inventory khi đặt hàng

Khi khách đặt món, code check product.IsAvailable nhưng không trừ inventory. Field inventory trong database hoàn toàn bị bỏ quên.

BUG-09: N+1 Query Problem trong OrderTracking

Trong OrderTracking.cshtml.cs, với mỗi order item, client gọi riêng lẻ API để lấy tên sản phẩm/combo → N+1 requests. Nếu đơn có 10 món = 10 HTTP calls bổ sung.

BUG-10: UpdateUser không hash password khi cập nhật

Dùng EntityState.Modified với toàn bộ entity → nếu client gửi password mới dạng plaintext, nó sẽ được lưu thẳng mà không hash BCrypt.

🔴 B. QR Code Generation & Validation Flow

flowchart LR A["Admin tạo/mở bàn"] --> B["API sinh qr_code_token\n(GUID + HMAC signing)"] B --> C["Tạo QR chứa URL\nhttps://domain/?tableId=3&token=abc"] C --> D["Khách quét QR"] D --> E{"API validate token\n+ check table status"} E -->|"Valid + Available/Taken"| F["Cho phép xem menu & đặt món"] E -->|"Invalid/Expired"| G["Từ chối truy cập"]

Đề xuất:

  • Mỗi khi Admin mở bàn → generate qr_code_token mới (GUID + timestamp + HMAC)
  • Token có thời hạn (hết phiên khách = bàn trở về available → token hết hiệu lực)
  • Client gửi tableId + token trong mọi request đặt hàng → API validate trước khi xử lý
  • Token rotate mỗi phiên → chống tái sử dụng QR cũ
