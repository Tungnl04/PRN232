1. Introduction

1.1 Purpose

FoodQR is a restaurant order management system powered by QR Code technology. Customers scan a QR code at their table to view the menu, place orders, track order status in real-time, and pay online via VNPay. The system supports waitstaff, kitchen staff, and administrators in managing day-to-day restaurant operations with real-time communication.

1.2 Scope

ComponentDetails
PlatformWeb Application (Responsive)
BackendASP.NET Core Web API (.NET 8)
FrontendASP.NET Razor Pages (.NET 8)
DatabaseSQL Server + Entity Framework Core

PaymentVNPay Sandbox (HMACSHA512)
SecurityJWT Bearer Token + BCrypt Password Hashing

1.3 Actors

ActorDescription
Guest (Customer)Scans QR code at the table to view the menu, place orders, track order status, and make payments. No registration required.
Kitchen (Chef)Views the order board, updates the status of each dish (Pending → Preparing → Ready).
Staff (Waiter)Manages tables, views order details, serves dishes, switches tables, cancels orders/items, merges bills, and confirms payments.
Admin (Manager)Manages products, categories, combos, tables, users, coupons, store configuration, and views revenue reports.
VNPay (External System)Online payment gateway. Receives payment URL requests and returns transaction callbacks.

7. Business Rules

IDRuleDescription
BR-01One Table = One BillEach physical table can have at most one active Order at any given time. Subsequent orders are appended to the existing bill.
BR-02QR Token LifecycleToken auto-regenerates (new GUID) when table transitions to Cleaning or Available. Previous QR links are immediately invalidated.
BR-03Cancel Only When PendingItems can only be cancelled when their status is Pending. Once the kitchen marks an item as Preparing or Ready, cancellation is blocked.
BR-04Inventory Refund on CancelWhen an item is cancelled, the product inventory is restored: product.Inventory += item.Quantity. When inventory reaches 0, the product is auto-disabled.
BR-05Anti-Duplicate PaymentIf PaymentStatus = Success, all subsequent payment requests are rejected. Prevents double-charging from page refresh.
BR-06Merge Bill Keeps TableWhen merging Bill A into Bill B, all OrderItems move to B. Bill A is marked as Merged. Table A retains its Taken status (customers still seated).
BR-07Soft Delete PolicyProducts, Categories, and Users are never hard-deleted. Instead, they are deactivated via IsAvailable = false or Active = false to preserve historical data integrity.
BR-08Forced Password ChangeNewly created accounts have MustChangePassword = true. Users must change their default password before accessing the dashboard.
BR-09Host Order PatternWhen multiple guests scan the same QR code, the first person to submit an order becomes the Host. All subsequent orders are appended to the Host's bill.
BR-10Auto-Disable Out of StockWhen Inventory <= 0, the system automatically sets IsAvailable = false and notifies the Admin.
BR-11Coupon Validation PipelineCoupons must pass 5 sequential checks: Exists → Is Active → Not Expired → Has Remaining Uses → Order meets minimum amount.
